# Security Constraints

nas のサンドボックス設計における不変条件。spec 作成時・コードレビュー時に違反がないか確認する。

## コンテナ境界

### C1: シークレットをコンテナにマウントしない

エージェント（コンテナ内プロセス）がシークレットの生値にファイルシステム経由でアクセスできてはならない。

- シークレットフレームファイル（`mask-secrets.frame`）はホスト専用ディレクトリに配置する
- hostexec の session broker dir、mask-filter 用のランタイムディレクトリなど、ブローカーだけが読むファイルはコンテナにマウントしない
- コンテナにマウントするのはマスク済み出力やバイナリツール（読み取り専用）に限る

### C2: control socket はコンテナに露出しない

hostexec の control socket（approve/deny/list_pending）はホスト側のみに公開する。コンテナにマウントされるのは exec socket（execute/fallback のみ）。これにより悪意あるエージェントが自身のリクエストを承認することを防ぐ。

### C3: ホスト実行の出力もマスク対象

hostexec 経由でホスト側で実行されたコマンドの stdout/stderr にもシークレットマスクを適用する。コンテナ内コマンドと同じレベルの保護を提供する。

## シークレット管理

### S1: シークレットの解決はホスト側プロセスのみ

`resolveMaskSecrets`、`SecretStore.require` などシークレットを平文に展開する処理はブローカーやステージ（ホスト側）でのみ行う。展開結果をコンテナ内環境変数やファイルとして渡さない（hostexec rule の env 注入は意図的な例外: ユーザーが明示設定したルールのみ）。

### S2: シークレットフレームの生存期間

シークレットフレームファイルはセッション終了時に確実に削除する。ブローカーの `close()` でクリーンアップする。

## ネットワーク

### N1: コンテナからホストへの通信は Unix socket 経由のみ

コンテナがホストサービスにアクセスする手段は明示的にマウントされた Unix socket に限定する。TCP でホストポートに直接接続させない。
