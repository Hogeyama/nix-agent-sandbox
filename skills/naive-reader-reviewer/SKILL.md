# Naive Reader Reviewer

ドキュメントを「対象ドメイン・プロダクト・コードベースを一度も見たことがない新規読者」の目線でレビューし、**知識の呪い**（書き手には自明でも読者には伝わらない箇所）を検出するワークフロー。

実体のレビューは、新規読者を再現した**サブエージェントを隔離起動**して行う。orchestrator（あなた）の役割はその起動を正しく行い、結果を提示することだけ。

## なぜ隔離が核心か

このレビューの価値は、レビュアーが **対象について何も知らない状態** で読むことにある。書き手や、書き手と会話してきたあなたは、すでに設計意図・背景・ソースコードを知っている＝「知識の呪い」にかかっている。だから「ここは説明しなくても分かるはず」という箇所を見落とす。それを見落とさないために、文脈ゼロの新規読者を別コンテキストで再現する。

したがって orchestrator の最重要責務は、**サブエージェントに余計な文脈を漏らさないこと**。これを破ると、サブエージェントも同じ呪いにかかり、レビューが無価値になる。

## 原則

- **判定はサブエージェントに任せる。** 「ここは伝わる／伝わらない」の判断を orchestrator 自身で代行しない。あなたは会話の文脈を知ってしまっているので、その判断は信用できない。
- **修正しない（このスキルの範囲では）。** 指摘リストを返すまでがこのスキル。Edit / Write でドキュメントを直すのは、結果を見たユーザーが望んだ場合に、別途行う。
- **対象ドキュメントは「パスで」渡す。** 本文をあなたが要約・抜粋して渡さない。サブエージェントが自分で Read して、フラットな初見で読むのが正しい。

## Step 1: レビュー対象を特定する

ユーザーの依頼・引数・会話から、レビュー対象のドキュメントのファイルパスを確定する。

- 引数や会話でパスが明示されていればそれを使う。
- 「さっき書いたドキュメント」のように曖昧なら、どのファイルかを 1 つ確認してから進む。憶測で対象を決めない。
- 複数ファイルが対象なら、関連する一群として 1 回の起動でまとめて渡してよい。数が多く独立性が高い場合は、ファイル単位で並列に起動してもよい。
- ユーザーが「これは前提として読んでよい」とドキュメントを挙げたら、それを**前提ドキュメント**として控える。読者が既に読んでいる資料で、レビュアーはこれを読んで「読者が既に知っていること」のベースラインにする。

## Step 2: 読者スペックを確定する

**読者スペックはレビュー品質を左右する最重要入力**。次を含む形で定める。

- **誰が読むか**（例: 入社初日のバックエンドエンジニア）
- **読了後に何ができるべきか**（例: 開発環境を一人で構築できる）
- **何を既知とするか** — 一般知識に加え、**読者が既に読んでいる前提ドキュメント**で表す。前提ドキュメントが無ければ簡潔な散文で（例: Docker は既知、社内ツールは未知）。

このうち「読了後に何ができるべきか」は単なる背景情報ではなく、**レビューの判定基準そのもの**。レビュアーは各指摘を「このゴールを止めるか否か」で仕分けるので、ゴールが曖昧だと仕分けも曖昧になる。検証可能な行動（〜できる）で書くこと。

決め方:

- ユーザーの読者像（「初見のシニアエンジニア」のような一言でもよい）を出発点に、「誰が読むか」「読了後に何ができるべきか」を短く草案する。
- **対象固有の「既知」は、用語を手で列挙せず、前提ドキュメントのパスで表す。** 「intro と quickstart は読んでいる」と前提ドキュメントを渡せば、レビュアーがそれらを読んで既知ベースラインにする。長大な既知/未知リストを作るのではなく、前提ドキュメントにその役割を担わせる。
- 一般知識（一般的なソフトウェア開発の常識、広く知られた概念・用語、読者が LLM エージェントなら実行環境の一般的な仕組み）は黙って「既知」でよい。「未知」は基本「前提ドキュメントにも一般知識にも無い、対象プロダクト固有の事柄」で、暗黙でよい。
- 読者像も前提ドキュメントも無いときはデフォルト（対象ドメイン・プロダクト固有の知識はゼロ、一般常識と一般的なソフトウェア開発の知識は持つ読者）から始める。

## Step 3: 新規読者レビュアーを隔離起動する

`Agent` ツールでレビュアーを起動する。

- `naive-reader-reviewer` が subagent_type として登録されている環境では、それを使う。
- 登録されていなければ、**このスキル同梱の `<skill-dir>/agents/naive-reader-reviewer.md` を読み、その全文をプロンプト先頭に貼って general-purpose agent を起動する**。

起動プロンプトに **含めてよいのは次のものだけ**:

1. レビュー対象ドキュメントの**ファイルパス**（複数可）
2. **前提ドキュメントのファイルパス**（あれば。読者が既読の資料として、レビュアーが読んでよいもの）
3. Step 2 で確定した**読者スペック**（誰 / ゴール / 特記すべき既知・未知。多くは前提ドキュメントで代替でき、短くてよい）

指摘の型と出力フォーマットの定義はペルソナ（`subagent_type` または同梱 `agents/naive-reader-reviewer.md`）に任せる。起動プロンプトは上記 3 つの提供に徹すれば、5 つの型（未定義語 / 宙づりの参照 / 根拠のない参照 / 論理の飛躍 / 未充足の読者ゴール）と出力形式がブレない。

**含めてはいけないもの（漏らすと隔離が壊れる）**:

- これまでの会話・依頼の経緯
- ドキュメントの設計意図・背景・「本当はこういうことが言いたい」という補足
- 関連するソースコード・スキーマ・設計ドキュメントへのポインタ
- あなた自身による本文の要約・言い換え・「ここはこういう意味」という注釈

つまり「ファイルパスと読者スペックだけを書いた、文脈ゼロの依頼文」を渡す。サブエージェントが白紙の状態でファイルを開いて読むのが正しい姿。

### 起動前に、依頼文をユーザーに見せる

組み立てた依頼文（＝確定した**読者スペック**と**対象ファイルパス**）を、**起動する前にそのままユーザーに提示する。**

- 見せる読者スペックは簡潔に保つ（誰 / ゴール / 前提ドキュメントのパス）。
- 指摘がなければそのまま起動する。修正が入ったら反映して起動する。
- ペルソナ定義（同梱 `agents/naive-reader-reviewer.md`）の全文は定型なので、見せるのはあなたが組み立てた読者スペックと対象パスの部分でよい。

## Step 4: 結果を提示する

サブエージェントが返した指摘リストを、ユーザーに提示する。

- **指摘は要約・取捨選択せずにそのまま渡す。** あなたが「これは実際には伝わるから落とそう」と判断するのは、まさに呪いにかかった判断。落としたい指摘があってもユーザーに見せた上で委ねる。
- レビュアーは指摘を「ゴールを止める」「参考（止めない）」に仕分けて返す。この仕分けも改変せずそのまま見せる。修正に進む場合に直す価値があるのは原則「ゴールを止める」側だけ（その判断は doc-clarity-loop / ユーザーが行う）。
- どの読者スペックでレビューしたかを添える（Step 2 でデフォルトを使った場合は特に）。
- 提示の最後に、ユーザーが望めば指摘に基づいてドキュメントを修正できる旨を一言添えてよい。ただし**自動では直さない** — 直すかどうか・どう直すかはユーザーの判断。

## Red flags

- 対象ドキュメントの本文を orchestrator が要約してサブエージェントに渡す（白紙性が壊れる）
- 起動プロンプトに会話の経緯・設計意図・ソースコードを混ぜる（呪いが伝染する）
- 読者スペックを確認せず、勝手に「上級者向け」「初心者向け」と決めつける
- サブエージェントの指摘を orchestrator が「これは伝わるはず」と間引いてからユーザーに見せる
- 指摘リストの提示を飛ばして、いきなりドキュメントを書き直し始める
