# Doc Clarity Loop

ドキュメントを、新規読者に伝わる状態へ反復改善するワークフロー。
**新規読者レビュー → 致命的な指摘を修正 → 新規読者で再レビュー** を、ユーザーの承認を挟みながら回す。

検出だけが目的なら、このループは要らない —— `naive-reader-reviewer` を単発で使う。このスキルは「直すところまで」やりたいときのもの。

このループには既知の失敗モードがある: 新規読者は「分からなかった所」しか報告しないため、指摘を全部真に受けて直すと**説明が単調増加し、ドキュメントが冗長化する**。これを防ぐ仕組みが、修正前のトリアージ（ゴールを止める指摘だけ直す）と、再レビュー時の簡潔さレビューの 2 つ。どちらも省略しない。

## なぜ役割を分けるか

このループの肝は **役割を厳密に分けること**。混ぜると価値が壊れる。

| 役 | 文脈 | 仕事 |
|---|---|---|
| **reviewer** | 文脈ゼロ（隔離） | 読者ゴールを止める箇所を見つける（`naive-reader-reviewer` の作法） |
| **fixer** | **フル文脈**（ソース・設計・会話あり） | 指定された指摘を、事実として正しく直す |
| **editor** | 対象ドキュメントと diff | 修正で増えた冗長さ・重複・くどさを指摘する（足し算に対する引き算役） |
| **orchestrator（あなた）** | 進行管理 | ループを回し、トリアージ・ユーザーゲート・停止判断を担う |

- **fixer は reviewer の真逆。** 直すにはドメイン知識が要るので、fixer はソース・設計を読んでよい（むしろ読んで裏取りすべき）。reviewer が絶対に持ってはいけない文脈を、fixer は持つ。
- **再レビューは毎回まっさらな新規読者を起動する。** 同じレビュアーを使い回さない。新しい文脈ゼロの読者が、直った版を初見で読むからこそ隔離が保たれる。
- **orchestrator はレビューも修正もしない。** 「ここは伝わる/伝わらない」は reviewer、本文の編集は fixer。あなたは進行とユーザー合意だけを担う。

## 原則

- **実ファイルを編集する前に、ユーザーの合意を得る。** fixer は対象ドキュメントを直接書き換える。どのファイルを対象にし、どの指摘を直すかを提示し、承認を得てから fixer を起動する。
- **指摘ゼロまで回さない。** 文脈ゼロの新規読者は、十分良いドキュメントにも必ず何か見つける。停止条件は「**残った指摘が読者ゴールを止めない状態になったか**」であって「指摘が 0 件になったか」ではない。
- **指摘は証拠であって修正命令ではない。** 新規読者の指摘は「その読者がそこでつまずいた」という観測結果で、直すべきかの判定は別。読者スペックの**ゴール（読了後にできること）を実際に止める指摘だけ**を直す。知らない語があるだけでゴールに支障がないものは「直さない」と決めて記録する。
- **1 ラウンド最大 3 件。埋める義務はない。** ゴールを止める指摘が 1 件ならその 1 件だけ直す。0 件ならループを終える。
- **fixer が捏造したら止める。** fixer が確証なく挙動・既定値・安全性を断定していないか、diff で確認する。確証の持てない箇所は「未解決」として残し、author（ユーザー）に確認する。

## Step 1: セットアップ

1. **対象ドキュメント**・**前提ドキュメント**・**読者スペック**を確定する。やり方は `naive-reader-reviewer` の Step 1–2 と同じ（読者スペックは「誰 / ゴール / 既知（前提ドキュメントで表す）」を短く草案し、起動前にユーザーへ見せて直してもらう）。
2. 対象ドキュメントが git 管理下なら、開始時点の `git status --short` を控える。fixer の変更を後で diff で見せ、revert できるようにするため。**勝手にコミットしない。**
3. ループ状態を控える: `round`（ラウンド番号）、`target_findings`（今回直す指摘）、`max_rounds`（既定 2〜3）。

## Step 2: 新規読者レビュー

reviewer を起動して、対象ドキュメントの指摘リストと「最も致命的な3件」を得る。起動の作法（含めてよいのは対象パス＋前提ドキュメントパス＋短い読者スペックだけ／会話や設計意図を漏らさない）は `naive-reader-reviewer` に従う。

返ってきた指摘は、要約・取捨選択せずユーザーに提示する。

## Step 3: トリアージして修正対象を決める

1. レビュアーの指摘を「**ゴールを止める**」（手順が再現できない / 誤った行動に導かれる / 記述が矛盾して見える）と「**ゴールは止めない**」（知らない語だが読み飛ばせる、あれば親切な補足、本文が要求していない手順への杞憂）に分ける。レビュアーが指摘ごとに付けた「何ができなくなるか」のタグを判定材料にする。タグが仮定形（「〜と不安に思うかもしれない」）の指摘は止めない側に落とす。
2. ゴールを止める指摘から最大 3 件を `target_findings` とする。止めない指摘は直さず、スコープ外として記録する。
3. ユーザーに「この N 件を直し、これらは直しません」と両方提示し、**承認を得る**（範囲の増減・優先の指示があれば反映）。
4. ここで初めて実ファイル編集に進む合意が取れる。`target_findings` が 0 件なら編集せず Step 6 へ。

## Step 4: fixer を起動する

fixer に次を渡して起動する。

- 対象ドキュメントのパス
- `target_findings`（今回直す指摘の全文）
- 読者スペック
- 背景情報の所在（リポジトリのパスなど。fixer はソース・設計を読んで裏取りしてよい）

fixer は指摘ごとに応答パレット（リンク / 書き換え / 追記 / 対象外差し戻し。追記は最終手段 —— 詳細は `agents/fixer.md`）から手段を選んで編集し、「解消した指摘」「未解決の指摘（理由）」「対象外として差し戻した指摘」を返す。

## Step 5: diff を見せて再レビュー

1. `git diff`（または編集前後の差分）をユーザーに提示する。fixer が確証なく事実を断定していないか、スコープ外を壊していないかを一緒に確認する。
2. 2 種類のレビューを行う（並列でよい）。
   - **新しい reviewer を起動して再レビューする**（Step 2 と同じ作法、ただし対象は更新後のファイル）。これは使い回しでなく、毎回まっさらな新規読者。
   - **editor（簡潔さレビュアー）を起動する。** 対象ドキュメントと今回の diff を渡し、修正で増えた冗長さ・重複・括弧書き定義を指摘させる。
3. 再レビュー結果で、`target_findings` が解消されたかを確認する。新たに浮上した「ゴールを止める」指摘と、editor の指摘があれば提示する。
4. editor の指摘は品質の後始末であり、**ループ継続の判定（Step 6）には数えない**。次ラウンドへ進むならそのラウンドの fixer に同梱する。今ラウンドで終了するなら、ユーザーの承認を得て終了前に 1 回だけ fixer で直す（この修正への再レビューは不要）。

## Step 6: 継続するか止めるか

- **`target_findings` が解消され、残った指摘がゴールを止めない** → ループ終了。最終差分と、スコープ外として残した指摘を一覧で報告する。
- **再レビューで `target_findings` の解消が確認できない / 新たに「ゴールを止める」指摘が出た** → 次ラウンドの修正対象をユーザーと合意し、Step 4 へ戻る。
- **fixer が「未解決（author 確認が必要）」を返した** → その指摘はループで潰さず、何を確認すれば直せるかをユーザーに渡して判断を仰ぐ。
- **`max_rounds` 到達、または同じ指摘が解消されないまま繰り返す** → 惰性で回さず、現状の差分・残課題・ユーザーに決めてほしい点を整理して止める。

## Red flags

- ユーザーの承認なしに fixer が実ファイルを書き換える
- レビュアーの指摘を全部直そうとする（トリアージを飛ばすと説明が単調増加し、ドキュメントが冗長化する）
- fixer の修正が「初出に括弧書きで定義を挿入」ばかりになっている（応答パレットが機能していない兆候）
- 簡潔さレビュー（editor）を省略してラウンドを重ねる
- 指摘が 0 件になるまで機械的にループを回し続ける
- 再レビューで同じレビュアーを使い回す（隔離が壊れ、初見でなくなる）
- fixer に会話の経緯や設計意図を渡しておきながら、同じプロンプトでそのまま reviewer も起動してしまう（reviewer は常に文脈ゼロ）
- fixer が確証なく挙動・既定値・安全性を断定した本文を、確認せずに採用する
- orchestrator が自分でレビュー判定や本文編集をしてしまう
