# Anthropic egress スキーマ認識マスク設計

## 背景と問題

例えば Git リポジトリにシークレットが平文で混入している状況を考える。この前提で「シークレットを Anthropic（api.anthropic.com）へ送信しない」ことを保証したい。

既存の対策は3つ存在する。

1. `MaskConfig.maskfs`（FUSE でワークスペースをマスク）: git diff が常時発生して使用に耐えなかった
2. `MaskConfig.proxy`（mitmproxy でリクエストをマスク）: バイト列パターンの総当たり置換のため、アプリ層エンコード（JSON 内 base64 blob、gzip-in-JSON など）を取りこぼす
3. `MaskConfig.filter`（Bash 出力をマスク）: Bash stdout のみをラップする

**未解決のギャップ**: エージェント（コンテナ内の Claude Code）は Read ツールやプロンプト内の `@file` 展開経由でシークレットを読み取る。`.filter` は Bash stdout しか捕捉しないため、これらは素通りして API リクエストボディに載る。

## 核心的な再フレーミング

エージェントが何を経由してシークレットを読んでも（Bash / Read / `@file` / MCP / サブエージェント）、Anthropic に届くには必ず api.anthropic.com への egress を通過する。したがって **egress だけが全経路を一点で捕捉できるチョークポイント**に該当する。取り込み側（ingestion）のマスクは、Read・`@file`・MCP・将来機能を延々と追いかける「もぐら叩き」に帰結し、完全性を担保できない。とりわけ `@file` はツール呼び出しではなくクライアント側の展開のため、フックが介入できない。

本設計は egress の proxy を「バイト列パターンのブロックリスト」から「**API スキーマのアローリスト**」へ転換する。宛先の契約（Messages API のスキーマ）を把握しているため、平文が入りうる場所を漏れなく列挙してデコードでき、理解できない構造をフェイルクローズドで弾ける。

## 目標

- api.anthropic.com へのリクエストボディに含まれる既知シークレット値を、エンコード変種（base64・URL エンコード・gzip）を再帰的にデコードした上で捕捉し、`****` へ置換する。
- 理解できない構造（未知パス・未知コンテンツブロック型・未知ベータ・デコード不能なコンテンツ）はフェイルクローズドで遮断または人間判断へ回す。
- API スキーマの進化に対し、維持コストを「新しい `anthropic-beta` 値の登場」と「新しいコンテンツブロック型の登場」の2種に限局する。

## 非目標

- **ターンをまたぐ分割送信**（「前半 ABC」「後半 DEF」を別リクエストに分割）の検知。リクエスト単位の値照合では原理的に検知できず、本スコープ外とする。
- **モデルによる言い換え・要約**の検知。値照合の範囲外とする。
- **api.anthropic.com 以外のホスト**へのマスク。第三者への exfiltrate は別脅威であり、既存の allowlist と汎用マスクが引き続き担当する。本設計はスコープを Anthropic API ホストに限定する。
- **レスポンスボディ**の処理。漏洩ベクトルは egress リクエストであり、レスポンスは対象外とする。

## 戦略の要点: 検知したらマスク、理解できなければ遮断

2種類の挙動を明確に分離する。

- **理解できるフィールド内で既知シークレット値を検知**した場合 → `****` へマスクしてリクエストを継続する。
- **構造上の未知**（未知パス・未知コンテンツブロック型・未知ベータ・宣言コーデックでデコードできないコンテンツ）に遭遇した場合 → フェイルクローズドで遮断（403）または broker の人間判断へ回す。マスクの完全性を保証できない箇所は、通過させずに拒否する。

マスクの取りこぼしがそのまま漏洩になる現行 proxy と異なり、理解できない箇所を必ず遮断側へ倒すことで、漏洩をブロックへ変換する。

フェイルクローズド時の既定挙動は「broker の人間判断チャネルが利用可能なら人間判断へ回し、利用不可（ヘッドレス等）なら 403 で遮断する」ものとする。この既定は設定で上書き可能とする。

## アーキテクチャ

### スキーマ変種の選択（バージョニングの実態を利用）

Anthropic の `anthropic-version` ヘッダーは約3年間 `2023-06-01` に固定されており、同一バージョン内では追加のみを保証する（既存入出力パラメータの削除・変更をしない）。破壊的変更は `anthropic-version` ではなく **モデルID** と **`anthropic-beta` ヘッダー** の2軸に乗る。

`anthropic-beta` ヘッダーは「このリクエストが標準外のどのフィールド／ブロックを使うか」を Anthropic 自身が列挙した宣言に等しい（例: `files-api-2025-04-14` があれば `file_id` source ブロックが出現しうる）。これを利用し、proxy のスキーマ表を次の構造で保持する。

- **基底スキーマ**（`anthropic-version: 2023-06-01` に対応）
- **`anthropic-beta` 値ごとのデルタ**（各フラグが解禁する追加フィールド・ブロック型・コーデックを定義）

proxy はリクエストの `anthropic-version` と `anthropic-beta` を読み、有効スキーマを合成する。**認識できない `anthropic-beta` 値が付いていること自体が、モデル化できない機能を使うリクエストのフェイルクローズド信号**に該当する。

### 処理パイプライン

api.anthropic.com 宛のリクエストに対してのみ、以下を適用する。

1. **ルーティング**: `(method, path)` でハンドラを選択する。既知パス（`/v1/messages`、`/v1/messages/count_tokens`、`/v1/files` など）はスキーマハンドラへ、未知パスはフェイルクローズドへ回す。
2. **ベータ検証**: `anthropic-beta` の各値を認識可能か判定する。未知の値が存在すればフェイルクローズドへ回す。
3. **正規化**: 各既知フィールドを有効スキーマの宣言に従ってデコードする。平文フィールドはそのまま、image/document/`file_id` などの base64 source はバイト列へ展開する。
4. **マスク**: 正規化後の各所に既知シークレット値を照合し、一致箇所を `****` へ置換する。値照合は既存の `mask_patterns`（base64 確定部分文字列展開を含む）を流用し、対象を生バイト列ではなく正規化済みフィールド値へ変更する。
5. **再直列化**: マスク済みの値で JSON を組み直して上流へ送出する。

### フィールド単位の扱い

**第1層: 既知フィールドの表（手動維持）**

エンドポイントごとに JSON パス単位で `{ 種別: content|control, コーデック: plaintext|base64|json|... }` を宣言する。これは公開 API スキーマから著述するもので、実行時推論ではない。

- `messages[].content[].text` → content / plaintext
- `messages[].content[].source.data`（image/document）→ content / base64
- `system` → content / plaintext
- `tools[].description` → content / plaintext
- `model`、`max_tokens`、`stream`、`metadata.user_id` → control / plaintext

**第2層: 未知フィールドの既定（位置スコープ）**

表に無いフィールドは content か control か判定できない。使える唯一の信号は値ではなく構造上の位置である。コンテンツは既知のコンテナ（`system`、`messages[].content`、`tool_result` の中身、`tools`）の内側にしか出現しないという事実を根拠に、既定を位置でスコープする。

- **コンテンツコンテナの内側の未知キー、または未知の `type` を持つブロック** → フェイルクローズドで遮断する
- **トップレベルの未知スカラ／メタデータ** → 削除せず平文としてマスク走査だけ掛けて通過させる（`cache_control`・`betas` など新パラメータの追加で壊さない）

### 挙動マトリクス

| 状況 | 挙動 |
| --- | --- |
| 既知パス + content フィールド（表に有り） | 正規化デコード → 値照合 → マスク → 継続 |
| 既知パス + control フィールド（表に有り） | 平文走査 → 一致すればマスク → 継続 |
| コンテンツコンテナ内の未知キー / 未知ブロック型 | フェイルクローズド（403 / 人間判断） |
| トップレベルの未知スカラ・メタデータ | 平文走査 + 通過 |
| 宣言コーデックでデコード不能なコンテンツ | フェイルクローズド（既存 `flow.mask_blocked` 機構） |
| 未知パス | フェイルクローズド |
| 認識できない `anthropic-beta` 値 | フェイルクローズド |

## 既存コンポーネントとの統合

- **`src/docker/mitmproxy/nas_addon.py`**: api.anthropic.com 宛リクエストに対して、汎用の `_apply_request_masking`（バイト列総当たり）に代えてスキーマ認識パスを適用する。他ホスト宛は従来どおり `_apply_request_masking` を使う。
- **フェイルクローズド機構**: 既存の `flow.mask_blocked = True` → 403 を再利用する。人間判断へ回す場合は broker の authorize/review フローへ接続する。
- **値照合**: 既存の `mask_patterns`（`_build_patterns` / base64 確定部分文字列）を流用する。生バイト列ではなく正規化済みフィールド値へ適用する点のみ変更する。
- **人間判断**: 既存の broker approve/deny UI と review rules 機構へ接続する。フェイルクローズド時の判断チャネルとして利用する。
- **セキュリティ制約**: シークレット値の解決はホスト側（broker/ステージ）で行い、addon へはセッションファイル経由でパターンを渡す既存設計を維持する（S1）。addon はマスク済み出力のみをコンテナ由来トラフィックへ返す。

## Config

本機能は Claude（api.anthropic.com）専用の保護であり、他エージェント（copilot / codex）の egress は別エンドポイントのため保護しない。有効化したまま非 Claude エージェントを指定すると「保護しているつもりで無防備」になるため、設定検証で明示的にエラーへ倒す。

### フラグ

`MaskConfig` に opt-in フラグ `anthropicEgress: boolean` を追加する。**デフォルトは `false`**（`Schema.pkl` で `anthropicEgress: Boolean = false`、`types.ts` の `MaskConfig` に同名フィールド）。true のときのみ、addon が api.anthropic.com 宛にスキーマ認識マスクを適用する。false/未設定なら addon は現行の汎用 `_apply_request_masking` を Anthropic ホストにも適用する（現行挙動を維持）。

デフォルトを false にする理由は2つある。ユーザーの利用形態（フラグを立てて有効化）に一致すること。そして下記のエラー検証と両立すること（デフォルト true にすると非 Claude の既定実行がすべてエラーになる）。

### 検証（`validate.ts`、プロファイル単位）

- `mask.anthropicEgress` の boolean 型チェック。
- `anthropicEgress === true` かつ `agent !== "claude"` → エラー。この保護は api.anthropic.com のトラフィックのみを対象とするため。
- `anthropicEgress === true` かつ `mask.proxy !== true` → エラー。スキーママスクは mitmproxy addon 上で動作するため proxy を要求する。

### addon へのゲート伝達

`anthropicEgress` を config → proxy stage → `SessionBrokerConfig` → セッションレジストリ（`SessionRegistryEntry.anthropicEgress?`）の順で伝達する（`agent` フィールドと同じ経路）。addon は既存の `_load_registry` でレジストリを読み、フラグが true のときのみスキーマパスを発動する。この伝達に関わる proxy stage / broker service の変更は effect-separation に従う。

## スキーマ表の維持

- 表は基底スキーマ + `anthropic-beta` 別デルタで構成し、リポジトリ内でバージョン管理する。
- 追随トリガーは2種に限局する: 新しい `anthropic-beta` 値の登場、新しいコンテンツブロック型の登場。制御・メタデータフィールドの追加（変更頻度が最も高い層）は第2層の位置スコープ既定が破壊なく吸収するため、追随を要さない。
- **可観測性**: コンテンツコンテナ内でフェイルクローズド既定が発動した際、および未知ベータで遮断した際は、必ず log + アラートを出す。新ブロック型・新ベータの出荷当日に気づけるようにする。

## 残余リスク

- ターンをまたぐシークレット分割送信は検知できない（非目標）。
- 新しいコンテンツブロック型・新しいベータの出荷当日は、スキーマ表が追いつくまでフェイルクローズドが発動してセッションが中断しうる。可観測性のアラートで検知し、差分更新で復旧する。

## テスト方針

- Unit: スキーマ表のルーティング、フィールド分類、正規化デコード（base64/URL/gzip）、位置スコープ既定の分岐、ベータ合成ロジック。
- Integration: 実際の `/v1/messages` リクエスト形状に対し、text/image(base64)/document/tool_result 各所に埋めた既知シークレットがマスクされること、未知ブロック型・未知ベータ・未知パスが遮断されること、制御フィールド追加が通過すること。
- 既存の `nas_addon` integration test（mitmproxy 実プロセス）を拡張する。
