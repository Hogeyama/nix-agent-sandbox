# ワークスペース文字列マスク(maskfs)設計

日付: 2026-07-02
ステータス: 承認済み(実装計画待ち)

## 目的

ワークスペース内のファイルに含まれる特定の文字列(パスワード・APIキー等)を、コンテナ内で動く AI agent から秘匿する。agent がどの経路でファイルを読んでも(Read/Grep ツール、Bash の `cat`、ビルドプロセス等)、秘密値は同一長の `*` に置換されて見える。

### 要件(確認済み)

- **秘匿優先**: 表示上のマスクではなく、秘密のバイト列がコンテナ内に存在しないこと。LLM API へ秘密値が送信される経路を塞ぐ
- **実行時に本物の値は不要**: コンテナ内でのテスト実行やサーバ起動に秘密値は使わない。完全に置換してよい
- **書き込みポリシーは設定で選択可能**: `readonly`(秘密値を含むファイルへの書き込みを拒否)/ `passthrough`(無加工で通す)
- **同一長マスク**: `passw0rd` → `********`(8文字)。ファイルサイズが不変になり、FUSE 実装が単純・堅牢になる
- **秘密値の取得元は既存 Secret ソースを流用**: `env:` / `file:` / `dotenv:` / `keyring:`(hostexec の `SecretConfig` と同形式)。リテラル指定は提供しない(config.pkl がコミットされると git 履歴経由で漏れるため)

## アーキテクチャ

ホスト側で動く Zig 製の FUSE パススルーファイルシステム(**maskfs**)を新設する。

```
ホスト実体 /repo ──(maskfs: 読み取り時に置換)──> $SESSION/maskfs ──(docker -v)──> コンテナ内 /repo
```

- ワークスペースの実体をソースディレクトリとして、読み取り時に秘密文字列を同一長の `*` に置換したビューを `$SESSION/maskfs` にマウントする
- Docker には従来のワークスペース実パスの代わりに maskfs マウントポイントをバインドする。コンテナ内パスは従来どおり実パス(host path == container path の見かけを維持)
- デーモンはホスト側プロセスなので、agent からは観測・干渉できない
- Zig は既に flake.nix のビルドパイプラインにある(`src/hostexec/intercept/` の execve シムと同じパターン)。`src/maskfs/`(`build.zig` + `maskfs.zig`、libfuse3 リンク)を追加する

## 設定(Pkl)

`Profile` に `mask` フィールドを追加する。

```pkl
mask {
  values {
    new { source = "dotenv:.env#DB_PASSWORD" }
    new { source = "env:MY_API_KEY" }
    new { source = "keyring:myapp/token" }
  }
  writePolicy = "readonly"   // "readonly"(デフォルト) | "passthrough"
}
```

- `src/config/Schema.pkl` / `src/config/types.ts` / `src/config/validate.ts` を拡張する
- hostexec のシークレット解決ロジック(`env:` / `file:` / `dotenv:` / `keyring:` の解決)は共有モジュールに切り出して両者から利用する
- バリデーション: 解決後の値が **4 文字未満ならエラー**(短い値は誤マスクを量産するため)
- `mask.values` が空または未設定の場合、maskfs は起動せず従来どおり素のバインドマウントを使う(既存ユーザーへの性能影響ゼロ)

## FUSE デーモンの挙動

| 操作 | 挙動 |
|---|---|
| `getattr` / `readdir` / 読み取り専用 `open` 等 | 純粋パススルー(同一長置換なのでサイズ・オフセットが不変) |
| `read(offset, size)` | 実体を前後 `maxSecretLen - 1` バイト拡張して読み、ウィンドウ内でバイト列置換して中央スライスを返す(チャンク境界を跨ぐ秘密値に対応) |
| 書き込みモードの `open` / `write` / `truncate` / `unlink` / `rename` | `passthrough`: 無加工で通す。`readonly`: 対象ファイルの現内容に秘密値を含む場合のみ `EROFS` で拒否(含有判定は inode + mtime でキャッシュ) |

- マッチングは正規表現なしの単純バイト列一致。複数秘密値に対応し、マッチしたバイトを 1 バイトずつ `*`(0x2A)に置換する。バイナリファイルにも同じ規則を適用する
- 秘密値のデーモンへの受け渡しは **stdin 経由**(argv は `/proc` に露出し、一時ファイルはディスクに残るため)
- マウントオプションに `allow_other` が必要(コンテナ内 uid からのアクセスを許可するため)。ホスト要件として `/etc/fuse.conf` の `user_allow_other`(NixOS は `programs.fuse.userAllowOther = true`)をドキュメント化し、未設定時は原因と対処を示すエラーで起動を中止する

## パイプライン統合

- 新ステージ `createMaskFsStage`(`src/stages/maskfs/stage.ts`)を **MountStage の直前**(`src/cli.ts` のパイプライン組み立て、現行 417 行付近)に挿入する
- ステージの責務:
  1. 秘密値の解決(共有シークレット解決モジュールを利用)
  2. maskfs デーモンの起動と mount ready の待機
  3. `PipelineState` に「マスク済みワークスペースルート」を記録
  4. セッション終了時のクリーンアップ(`fusermount3 -u` + SIGTERM)の登録
- `MountStage.planMount()`(`src/stages/mount/stage.ts`)は、マスク済みルートが state にあればバインドソースをそれに差し替える(コンテナ内ターゲットパスは従来どおり)
- **フェイルクローズ**: デーモン起動失敗・mount ready タイムアウト時はセッション起動を中止する。素のバインドマウントへのフォールバックはしない。セッション中にデーモンが死んだ場合、コンテナからワークスペースは見えなくなる(transport endpoint not connected)— 秘匿は保たれ、ユーザーはセッションを再起動する

## 既知の制限(ユーザー向けドキュメントに記載)

- **圧縮・エンコード済みデータは対象外**: `.git/objects` は zlib 圧縮されているため、秘密値が git 履歴にコミット済みの場合は `git show` 等で agent に見える。base64 エンコードやアーカイブ内の秘密値も同様。本機能は「秘密値がコミット履歴に含まれていない」ことを前提とする
- 環境変数やプロセスメモリ経由の露出はスコープ外(本機能はファイル読み取り経路のみを扱う)
- コンテナ起動後にホスト側で秘密値リストは再解決されない(セッション開始時点の値で固定)
- カーネルページキャッシュにはマスク済み内容が乗るが、置換は決定的なので整合性の問題はない

## テスト

- **Zig ユニットテスト**(`zig build test`): 置換ウィンドウアルゴリズム — チャンク境界跨ぎ、複数秘密値、マッチの重なり、バイナリ入力
- **TS ユニットテスト**(`bun test src/`): 設定パース/バリデーション(4 文字未満エラー含む)、シークレット解決モジュール、MaskFs ステージのプランニングと MountStage のソース差し替え
- **統合テスト**(`bun test tests/`、Docker 必須):
  - mask 設定ありでコンテナ起動 → コンテナ内 `cat` / `grep` でマスク済み内容を確認
  - `writePolicy = "readonly"` で秘密値含有ファイルへの書き込みが拒否されること
  - 秘密値を含まないファイルの読み書きが通常どおり動くこと
  - mask 設定なしのとき従来のバインドマウントが使われること

## 検討して不採用にした代替案

1. **RO オーバーレイ方式**: 起動時にワークスペースをスキャンし、秘密値を含むファイルだけマスク済みコピーを read-only バインドマウントで重ねる(`.nas/config.pkl` 保護と同じ既存パターン)。実装は最も単純だが、起動後に秘密値を含むようになったファイルを追跡できず、対象ファイルが編集不可になる。→ FUSE 方式が全アクセス経路・起動後の変更に対応できるため不採用
2. **エージェントフック方式**: Claude Code の PreToolUse/PostToolUse フックで Read/Grep のツール出力を書き換える。マウント変更不要で軽量だが、Bash の `cat` やビルドログが素通しになり秘匿要件を満たせない。agent 種別(Copilot/Codex)ごとに別実装も必要。→ 不採用
